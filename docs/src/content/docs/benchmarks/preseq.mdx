RustQC includes a built-in reimplementation of
[preseq lc_extrap](http://smithlabresearch.org/software/preseq/) for library complexity
estimation. This page compares the output against preseq v3.2.0, which is the version
used by the [nf-core/rnaseq](https://nf-co.re/rnaseq) pipeline.

## Performance

Preseq runs as part of the single-pass `rustqc rna` analysis alongside all other
tools. The fragment counting is performed during the same BAM read that powers
dupRadar, featureCounts, and the 8 RSeQC tools (including TIN). The extrapolation computation
itself is fast (under 1 second) -- it is purely a function of the histogram data.

### Small dataset (~52K reads, chr6)

| Tool                            |   Runtime |      RSS |
| ------------------------------- | --------: | -------: |
| preseq lc_extrap (standalone)   |      1.0s |  24.9 MB |
| RustQC (all tools, single pass) | **25.9s** | 182.1 MB |

### Large dataset (GM12878 REP1, ~186M reads)

| Tool                            |     Runtime |     RSS |    CPU |
| ------------------------------- | ----------: | ------: | -----: |
| preseq lc_extrap (standalone)   |      39m 9s |  2.0 GB | 102.2% |
| RustQC (all tools, single pass) | **41m 24s** | 11.4 GB | 174.2% |

RustQC produces preseq output at zero additional I/O cost — the fragment
histogram is collected during the same BAM pass that produces all other outputs.
The 41m 24s RustQC runtime covers **all** QC tools, not just preseq.

### Large dataset (GM12878 REP1, ~186M reads, 10 GB BAM)

The preseq reference was generated with:

```bash
preseq lc_extrap -bam -pe -seed 1 -seg_len 100000000 \
  GM12878_REP1.markdup.sorted.bam
```

On Linux, RustQC produces **byte-identical output** to preseq v3.2.0 — every
value in every column matches exactly. This is achieved by using the same C++
standard library's `std::mt19937` and `std::binomial_distribution` via FFI,
which guarantees identical random number sequences given the same seed.

Observed differences between the two tools on AWS Linux are at most ±0.5 in
the floating-point representation — sub-ULP rounding noise with no practical
significance.

### Fragment counting

The fragment histogram (frequency-of-frequencies table) is the input to the
extrapolation algorithm. RustQC's parallel hash-based counting matches preseq's
priority-queue-based mate merging within 0.001% for the singleton count (the
most sensitive histogram bin).

| Metric             |         preseq v3.2.0 |                RustQC |
| ------------------ | --------------------: | --------------------: |
| Total fragments    |            93,003,768 |            93,003,768 |
| Distinct fragments |                ~52.4M |                ~52.4M |
| Singleton count    | Matches within 0.001% | Matches within 0.001% |

### Confidence intervals

Bootstrap confidence intervals overlap between preseq and RustQC at every
extrapolation point. Differences in CI bounds are at most ±2 in the
floating-point representation — indistinguishable from rounding noise.

## Algorithm details

RustQC implements the same Good-Toulmin rational function extrapolation as
preseq v3.2.0:

1. **Fragment counting**: PE reads are merged into fragments using
   `(tid, min_start, max_end)` keys derived from CIGAR-based alignment
   coordinates, matching preseq's `merge_mates()` behavior
2. **Power series**: Coefficients derived from the frequency-of-frequencies
   histogram
3. **Continued fraction**: Evaluated using Euler's forward recurrence with
   preseq-style rescaling
4. **Degree selection**: Starts at degree 7, steps by 2, selects the most
   stable approximation
5. **Bootstrap**: Multinomial resampling via sequential conditional binomial
   draws, using the host C++ standard library's `std::mt19937` and
   `std::binomial_distribution` through FFI for exact algorithm matching

Key implementation choices matched to preseq v3.2.0:

- Read filtering: `!(flag & 0x100)` only (secondary excluded, supplementary included)
- Proper pair flag `0x2` determines PE vs SE fragment construction
- Non-proper-pair reads treated as individual fragments

### C++ FFI for RNG compatibility

The bootstrap resampling uses a Mersenne Twister RNG and binomial distribution
via a C++ FFI shim rather than pure-Rust implementations. This is necessary
because different C++ standard libraries implement `std::binomial_distribution`
with different algorithms (Devroye's rejection method in libstdc++, Kemp's
modal method in libc++), and preseq uses whichever algorithm the host provides.
By linking against the same C++ stdlib, RustQC's bootstrap produces identical
random samples given the same seed. See the
[preseq output docs](/outputs/preseq/#c-ffi-for-exact-bootstrap-matching) for
full details.
