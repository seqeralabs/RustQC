## Performance

RustQC's Qualimap analysis runs in the same single-pass BAM scan as all other
tools, adding negligible overhead to the total runtime. The original Qualimap
Java implementation requires a separate dedicated run on a **name-sorted** BAM
file, meaning a `samtools sort -n` step is typically needed before Qualimap can
run. RustQC works directly on coordinate-sorted BAMs (the standard output of
most aligners), eliminating that preprocessing step entirely.

| Tool | Runtime | Notes |
|------|---------|-------|
| `samtools sort -n` | ~5-10 min | Name-sort required by Qualimap Java (not needed by RustQC) |
| Qualimap rnaseq (Java) | ~10 min | Separate BAM pass, single-threaded, requires name-sorted input |
| RustQC (all tools) | ~4 min | Single pass, all analyses combined, coordinate-sorted input |

In a typical pipeline, the original Qualimap workflow requires **15-20 minutes**
(name-sort + Qualimap), while RustQC produces identical output with **zero
additional cost** beyond its shared BAM scan.

## Output comparison

### Coverage bias metrics

The coverage bias metrics from RustQC's Qualimap output match the original
Qualimap implementation:

| Metric | RustQC | Description |
|--------|--------|-------------|
| 5' bias | 1.20 | Ratio of 5' coverage to middle |
| 3' bias | 0.92 | Ratio of 3' coverage to middle |
| 5'-3' bias | 1.24 | Ratio of 5' to 3' coverage |

### Gene body coverage profile

RustQC produces a 100-bin coverage profile along normalized gene body positions
(5' to 3'), matching the format produced by Qualimap. Three tiers are generated:
total (all genes), high (top 500 by expression), and low (bottom 500).

The coverage profile format is identical to Qualimap's output and is directly
parseable by MultiQC's Qualimap rnaseq module.

### QC results file

The `rnaseq_qc_results.txt` file is formatted identically to Qualimap's output,
including:

- **Reads alignment** -- primary alignments, secondary, multi-mappers, gene
  assignment counts
- **Reads genomic origin** -- exonic, intronic, intergenic, and overlapping-exon
  read counts with percentages
- **Transcript coverage profile** -- 5'/3' bias metrics
- **SSP estimation** -- strand-specificity proportions

### Implementation details

RustQC replicates Qualimap's merged exon model (Picard Gene.Transcript) for
non-redundant per-gene coverage. This means overlapping exons from different
transcripts of the same gene are merged before computing coverage, avoiding
double-counting. The read enclosure check, strand-specificity estimation, and
junction motif counting all follow Qualimap's algorithms.

## Benchmark conditions

- **BAM file**: GM12878 ENCODE RNA-Seq, paired-end, ~185M reads, 10 GB
- **Annotation**: GENCODE GTF (used for both Qualimap Java and RustQC)
- **Qualimap version**: 2.3 (Java implementation)
- **RustQC**: 8 threads, `--gtf` mode
- **Hardware**: Apple M3 Max, 128 GB RAM
