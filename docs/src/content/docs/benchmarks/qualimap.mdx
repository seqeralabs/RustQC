## Performance

RustQC's Qualimap analysis runs in the same single-pass BAM scan as all other
tools, adding negligible overhead to the total runtime. The original Qualimap
Java implementation requires a separate dedicated run on a **name-sorted** BAM
file, meaning a `samtools sort -n` step is typically needed before Qualimap can
run. RustQC works directly on coordinate-sorted BAMs (the standard output of
most aligners), eliminating that preprocessing step entirely.

### Large dataset (~186M reads, GM12878)

| Tool                        | Runtime        | Peak RSS | Notes                                                          |
| --------------------------- | -------------- | -------- | -------------------------------------------------------------- |
| `samtools sort -n`          | 12m 16s        | 5.2 GB   | Name-sort required by Qualimap Java (not needed by RustQC)     |
| Qualimap rnaseq (Java)      | 49m 58s        | 4.9 GB   | Separate BAM pass, single-threaded, requires name-sorted input |
| **Total upstream pipeline** | **~1h 2m 14s** |          | Name-sort + Qualimap combined                                  |
| RustQC (all tools)          | 14m 54s        | 11.4 GB  | Single pass, all analyses combined, coordinate-sorted input    |

RustQC completes **all QC analyses** in under 42 minutes, compared to over
1 hour for the upstream Qualimap pipeline alone (name-sort + Qualimap).
Note that RustQC's 14m 54s covers every QC tool (dupRadar, featureCounts,
RSeQC including TIN, preseq, samtools, and Qualimap), not just the Qualimap equivalent.

### Small dataset (~52K reads, chr6)

| Tool                   | Runtime | Peak RSS | Notes                                         |
| ---------------------- | ------- | -------- | --------------------------------------------- |
| Qualimap rnaseq (Java) | 5.0s    | 704.6 MB | Separate BAM pass, requires name-sorted input |
| RustQC (all tools)     | 25.9s   | 182.1 MB | Single pass, all analyses combined            |

On the small dataset, RustQC uses only **182 MB** of memory vs Qualimap's
705 MB while running all QC tools in a single pass.

RustQC produces a 100-bin coverage profile along normalized gene body positions
(5' to 3'), matching the format produced by Qualimap. Three tiers are generated:

Qualimap Java. The total (all-genes) tier shows negligible floating-point
differences (&lt;0.03% relative) due to coverage aggregation order.

### QC results file

The `rnaseq_qc_results.txt` file is formatted identically to Qualimap's output
and is directly parseable by MultiQC's Qualimap rnaseq module, including:

- **Reads alignment** -- primary alignments, secondary, multi-mappers, gene
  assignment counts
- **Reads genomic origin** -- exonic, intronic, intergenic, and overlapping-exon
  read counts with percentages
- **Transcript coverage profile** -- 5'/3' bias metrics
- **Junction analysis** -- junction read counts and motif frequencies

### Implementation details

RustQC replicates Qualimap's merged exon model (Picard Gene.Transcript) for
non-redundant per-gene coverage. This means overlapping exons from different
transcripts of the same gene are merged before computing coverage, avoiding
double-counting. The read enclosure check, strand-specificity estimation, and
junction motif counting all follow Qualimap's algorithms.

## Benchmark conditions

- **Small dataset**: `test` sample, ~52K reads, chr6 only
- **Large dataset**: GM12878 ENCODE RNA-Seq, paired-end, ~186M reads
- **Run date**: 2026-03-09 (AWS cloud execution)
- **Annotation**: GENCODE v47 GTF (filtered to standard chromosomes, used for both tools)
- **Strandedness**: strand-specific-reverse (`--stranded 2` / `-p strand-specific-reverse`)
- **Qualimap version**: 2.3 (Java implementation, name-sorted BAM input)
- **RustQC**: coordinate-sorted BAM input, 174.2% CPU utilization on large dataset
- **Infrastructure**: AWS cloud (via Nextflow/Seqera Platform)
