RustQC reimplements 8 [RSeQC](https://rseqc.sourceforge.net/) tools — including TIN (Transcript
Integrity Number) analysis — as built-in analyses that run in a single pass over the BAM file. This page
compares the output of each tool against the originals, validated on two datasets from an AWS
cloud run (nf-core AWS megatests, 2026-03-07).

- **Small dataset**: `test.bam` — ~52K reads, chr6 subset
- **Large dataset**: `GM12878_REP1.markdup.sorted.bam` — ~186M reads (GM12878 ENCODE RNA-Seq)

Both upstream RSeQC and RustQC ran on the **same BAM file** in all cases.

## Performance

All RSeQC tools run together in the same single-pass BAM reading that also produces
dupRadar, featureCounts, preseq, samtools, and gene body coverage output. The timings
below show the original Python RSeQC tools run individually versus RustQC running
all tools in a single pass.

### Small dataset (~52K reads, chr6 subset)

| Tool                        |   Runtime |        RSS |
| --------------------------- | --------: | ---------: |
| bam_stat                    |      0.6s |     3.3 MB |
| infer_experiment            |      0.8s |     3.4 MB |
| read_distribution           |      1.9s |    94.4 MB |
| inner_distance              |      1.9s |    76.4 MB |
| junction_annotation         |      1.4s |    50.7 MB |
| junction_saturation         |      1.1s |     6.0 MB |
| read_duplication            |      1.3s |    56.4 MB |
| tin.py                      |    1m 35s |    45.5 MB |
| **RUSTQC_RNA (all-in-one)** | **25.9s** | **182 MB** |

### Large dataset (~186M reads, GM12878)

| Tool                        |     Runtime |         RSS |
| --------------------------- | ----------: | ----------: |
| bam_stat                    |      7m 08s |     50.4 MB |
| infer_experiment            |        5.7s |     72.5 MB |
| read_distribution           |      7m 35s |      1.1 GB |
| inner_distance              |      1m 11s |    883.2 MB |
| junction_annotation         |      6m 29s |    134.9 MB |
| junction_saturation         |      8m 19s |    951.8 MB |
| read_duplication            |     21m 53s |     10.2 GB |
| tin.py                      |  **FAILED** |           — |
| **RUSTQC_RNA (all-in-one)** | **18m 52s** | **13.8 GB** |

The sum of upstream RSeQC tools on the large dataset (excluding the failed TIN) is approximately
**53 minutes**. RustQC covers all of these plus samtools, Qualimap, dupRadar, featureCounts, and
preseq in a single **18m 52s** pass — roughly a **3× speedup** over RSeQC alone, with full
coverage of all tools. The upstream `tin.py` failed after 6+ hours and produced no output;
RustQC completes TIN as part of its single pass.

## infer_experiment

Infers the strandedness of an RNA-Seq experiment by sampling reads and comparing
their strand to annotated gene models.

### Small dataset — Identical

| Metric                               |  RSeQC | RustQC |
| ------------------------------------ | -----: | -----: |
| Failed to determine                  | 0.0775 | 0.0775 |
| Fraction sense (1++,1--,2+-,2-+)     | 0.0051 | 0.0051 |
| Fraction antisense (1+-,1-+,2++,2--) | 0.9174 | 0.9174 |

### Large dataset — Minor difference (sampling)

| Metric                               |  RSeQC | RustQC |
| ------------------------------------ | -----: | -----: |
| Failed to determine                  | 0.0670 | 0.1052 |
| Fraction sense (1++,1--,2+-,2-+)     | 0.0117 | 0.0185 |
| Fraction antisense (1+-,1-+,2++,2--) | 0.9213 | 0.8763 |

The difference is caused by a **read-sampling mismatch**: the upstream
`infer_experiment.py` defaults to sampling only 200,000 reads. On a 186M-read BAM,
this early-file subsample draws predominantly from the first few chromosomes, which
underrepresents loci where transcripts on both strands overlap — causing the
"failed-to-determine" fraction to appear lower than it truly is. RustQC processes
all reads, giving a more representative estimate.

Both tools correctly identify the library as **strongly reverse-stranded**
(antisense ~88–92%), so the practical interpretation is identical. When
`infer_experiment.py` is run with `-s 2000000000` (processing all reads), the
results are identical.

## TIN (Transcript Integrity Number)

RustQC reimplements RSeQC's `tin.py`, producing per-gene TIN scores that measure
transcript integrity via Shannon entropy of read coverage uniformity.

### Small dataset — Identical

| Metric      |   RSeQC |  RustQC |
| ----------- | ------: | ------: |
| TIN mean    | 52.7514 | 52.7514 |
| TIN median  | 55.5830 | 55.5830 |
| TIN std dev | 22.5333 | 22.5333 |

### Large dataset — RustQC only (upstream failed)

The upstream `tin.py` was terminated after **6+ hours** without producing output on
the large dataset. RustQC completes TIN analysis as part of its single-pass
BAM processing in under 19 minutes total (inclusive of all other analyses).

| Metric      |         RSeQC | RustQC |
| ----------- | ------------: | -----: |
| TIN mean    | — (timed out) | 68.287 |
| TIN median  | — (timed out) | 75.921 |
| TIN std dev | — (timed out) | 23.959 |

This is a key RustQC advantage: TIN completes reliably on large production datasets
where the Python tool is not practical.

### Output format

RustQC produces gene-level TIN scores (one row per gene, using the longest
transcript as representative), while RSeQC's `tin.py` produces transcript-level
scores. Both formats are compatible with MultiQC. The gene-level approach
provides a cleaner summary with one score per gene.

## Benchmark conditions

- **BAM file (small)**: chr6 subset test BAM, ~52K reads
- **BAM file (large)**: GM12878 ENCODE RNA-Seq, paired-end, ~186M reads
- **Annotation**: Ensembl GTF gene annotation (RustQC uses GTF directly; RSeQC uses BED12 derived from the same GTF)
- **RSeQC version**: 5.0.4
- **RustQC**: multi-threaded, `--gtf` mode for all tools
- **Infrastructure**: AWS cloud (nf-core AWS megatests, 2026-03-07)
